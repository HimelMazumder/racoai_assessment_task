# Project Report — E-Commerce Ordering & Payment System

## 1. Implementation Approach

### Technology Choices
We chose **Django + Django REST Framework** for this project because of its mature ORM, built-in admin panel, robust migration system, and first-class support for relational data modeling — all of which align directly with the assessment's requirements for OOP design, relational schemas, and clean REST APIs.

**PostgreSQL** was selected over SQLite for its support of row-level locking (`SELECT ... FOR UPDATE`), which is essential for the thread-safe stock reduction requirement. **Redis** was chosen for the category tree cache because of its sub-millisecond read latency and native support in Django's cache framework.

### Architecture
The project follows a layered architecture:
- **Models layer** — Django ORM classes defining the relational schema
- **Serializers layer** — DRF serializers handling validation and data transformation
- **Views layer** — API endpoints mapping HTTP requests to business logic
- **Services layer** — Pure Python functions encapsulating business rules (checkout, stock reduction, payment callbacks)
- **Strategies layer** — Abstract payment provider interface implementing the Strategy pattern

Each Django app (`users`, `products`, `orders`, `payments`) owns its own models, serializers, views, URLs, and services, keeping the codebase modular.

## 2. Rejected Alternatives

### Payment Architecture
We considered using a **single monolithic payment function** with `if/elif` branching for Stripe vs bKash. This was rejected in favor of the **Strategy Pattern** because:
- Adding a third provider (e.g., SSLCommerz) would require modifying the existing function, violating the Open/Closed Principle
- Each provider has different payload structures for both initiation and verification, making a unified function hard to maintain
- The Strategy pattern allows each provider's logic to be tested and deployed independently

### Stock Reduction Timing
We considered reducing stock **at checkout time** (before payment). This was rejected because:
- Users who abandon payment would lock up inventory
- It creates a reservation system that needs timeout/cleanup logic
- The assessment explicitly states "Stock is reduced after successful payment"

Instead, stock is reduced only after the webhook confirms payment success, with `select_for_update()` preventing race conditions.

### Category Tree Storage
We considered using **MPTT (Modified Preorder Tree Traversal)** for the category hierarchy. This was rejected because:
- The assessment explicitly requires **DFS algorithm implementation**, not a library
- MPTT adds complexity with `lft`/`rght` fields and requires careful management on insert/delete
- Our DFS + Redis cache approach is simpler, meets the requirement, and performs well with the Redis cache eliminating repeated traversals

## 3. Testing Approach & Results

### Unit/Service Tests (`apps/payments/tests.py`)
We focused testing on the most critical and complex business logic — the payment callback and stock reduction pipeline:

| Test | What It Verifies | Result |
|---|---|---|
| `test_successful_stock_reduction` | Stock decreases correctly when sufficient | ✅ Pass |
| `test_out_of_stock_boundary_raises_exception` | `OutOfStockError` raised when stock insufficient | ✅ Pass |
| `test_payment_success_callback_flow` | Webhook → order status `paid` → stock decremented | ✅ Pass |
| `test_stock_conflict_savepoint_rollback` | Race condition → `stock_conflict` status, payment stays `success` | ✅ Pass |

### API-Level Tests (`apps/users/tests.py`)

| Test | What It Verifies | Result |
|---|---|---|
| `test_user_registration_success` | Registration returns 201 with valid data | ✅ Pass |
| `test_user_registration_duplicate_email` | Duplicate email returns 400 | ✅ Pass |
| `test_login_returns_jwt_tokens` | Login returns access + refresh tokens | ✅ Pass |
| `test_profile_requires_auth` | Unauthenticated profile access returns 401 | ✅ Pass |

### Manual API Testing
All endpoints were verified via `curl` commands against the running Docker container:
- User registration with unique email constraint
- Product listing and admin-only write access
- Order creation with invalid product ID (400 response)
- Order creation with inactive product (400 response)
- Checkout with ownership verification (404 for non-owned orders)
- Full Stripe webhook callback flow

## 5. Final Verdict

The implementation satisfies all functional and non-functional requirements specified in the assessment:

- **User Management:** Registration, login (JWT), profile, email uniqueness — all working
- **Product Management:** Full CRUD with admin-only writes, public listing — all working
- **Order Management:** Create with items, list own orders (with payments), ownership isolation — all working
- **Payment System:** Strategy Pattern for Stripe/bKash, webhook handlers, mock payment flows — all working
- **Stock Reduction:** Thread-safe with `select_for_update()`, deadlock prevention, savepoint rollback for race conditions — all working and tested
- **DFS + Redis:** Recursive category tree traversal cached in Redis with 12-hour TTL — all working
- **OOP Design:** Django model classes, ABC strategy hierarchy, custom exceptions — all present
- **Data Validation:** Email uniqueness, product existence/status checks, quantity minimums, provider validation — all implemented
- **Admin Panel:** All 5 models registered with custom displays, filters, inlines — all working
- **Seeders:** Management command seeds admin user, categories, and products — working
- **Testing:** 4 service-level tests + 4 API-level tests, all passing
- **Documentation:** Swagger UI, ERD, architecture diagram, payment flow diagrams, technical reflections — all present
- **Docker:** PostgreSQL + Redis + Django all containerized and health-checked
